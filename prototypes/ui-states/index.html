tinyTypeless UI Prototype

一条很窄，但必须很顺的语音输入界面

这个原型只围绕 v1 黄金路径设计：打开应用后只有极轻的菜单栏面板和独立设置页；应用在后台时， 按下 Right Option 出现监听 HUD，抬起后 HUD 直接切到处理态，再快速淡出。

UI decisions
打开时
主面板只放状态和快捷入口，真正设置放到独立窗口，不把菜单栏面板做成控制中心。
按下时
弹出极小胶囊 HUD，不抢焦点，不显示复杂配置，只明确告诉用户“正在听”。
抬起后
同一个 HUD 原地形变为“处理中”，不要跳新窗口，不要二次确认，不要把结果展示成聊天气泡。
失败时
HUD 保留可操作信息，至少给“已复制 / 可重贴”的兜底，不让用户白说一遍。

1. 打开应用时的面板

建议只有两个前台形态：菜单栏弹出面板和独立设置窗口。菜单栏面板给状态和跳转，设置不要堆在 HUD 里，也不要塞进主流程。

2. 应用在后台时，按下和抬起怎么显示

推荐用同一个 HUD 做状态迁移。按下时出现“Listening”，抬起时直接切为“Processing”，成功后短暂确认并淡出。 下方按钮可以切换不同状态。

推荐节奏

1
按下

HUD 在 100ms 内出现，只说一件事：“正在听”。不要显示设置、日志、模型信息。

2
抬起

同一胶囊切到处理中，用户不用猜系统有没有收到结束信号。

3
成功

显示 Done 约 400 到 700ms 后淡出。不要再弹结果预览窗口。

4
失败

保留明确的 fallback 语义，例如“已复制，可手动粘贴”或“插入失败”。

当前建议是：后台不显示任何常驻悬浮条。只有在按下触发键之后才出现 HUD。 这比一直挂着一个浮窗更干净，也更像输入层而不是工作台。

3. 各个状态

v1 只需要很少的视觉状态。状态一定少，但每个状态都要让用户明确知道系统在做什么，以及失败时还能不能补救。

Idle / Hidden

后台静默 最轻

默认没有悬浮条，没有主窗。只有菜单栏图标和常驻后台进程。

Recording

按下触发键 立即反馈

小胶囊出现，标识 Listening。可以带一个轻微呼吸感，但不要做花哨波形。

ASR Processing

松键后 等待网络

胶囊原地切 Processing。不要换位置，不要闪烁，不要弹新面板。

Clean-up

文本整理 极短暂

视觉上可以继续沿用 Processing，不需要单独一个重状态，只要内部能分阶段就行。

Inserting

AX / paste 看不见更好

这个状态一般不必额外强调。对用户来说，理想体验是“字已经进去了”。

Success

Done 快速淡出

显示短暂成功确认即可，不要停留太久，不要让它挡住继续输入。

Fallback

AX 失败 仍要可用

如果走剪贴板回退，应明确告诉用户结果已经可粘贴，避免“看似成功其实没进去”。

Error

网络 / 权限 明确报错

错误文案要短，下一步动作要清楚，例如“麦克风未授权”或“网络失败，请重试”。

Permission

首次安装 非主路径

权限引导不要混进 HUD。建议通过菜单栏入口或独立设置页提示，避免污染日常使用路径。
