# voiceKey 工程权衡决策

最后更新：2026-04-11

## 1. 最终判断

`丝滑度第一，轻量第二。`

如果两者冲突，默认选更顺的方案。

这不是一句口号，而是当前项目的实际执行原则。

原因很简单：

- 这个工具是输入层
- 输入层每天会用很多次
- 用户会立刻感知到卡顿、误触、插入失败
- 用户不会因为架构更轻就原谅体验更差

所以后续所有技术决策都按这个顺序排：

1. 能不能更像 竞品
2. 能不能更稳
3. 能不能更快
4. 在不伤上面三条的前提下，能不能更轻

## 2. 不可妥协项

这 4 条不能被“轻量化”牺牲。

### 2.1 触发必须可靠

- 按下必须开始
- 松开必须结束
- 不能漏掉 down/up

### 2.2 松键后必须快

用户真正感知的是：

`松开键以后，多快看到最终文本进入当前输入框。`

### 2.3 插入必须稳

ASR 再强，只要贴不进去，这条链就是失败。

### 2.4 文本不能白说

任何失败都不能让用户整段重说。

## 3. 当前保留的轻量方案

这些地方继续保持轻量。

### 3.1 宿主与 UI

- `Swift + AppKit`
- `SwiftUI` 只用于少量设置页

这已经足够轻，也不会明显伤手感。

### 3.2 本地状态

- `settings.json`
- `sessions.jsonl`
- `Keychain`

现在不上数据库，不影响主路径体验。

### 3.3 网络层

- `URLSession`

只要连接复用做对，不需要引入更重的网络框架。

## 4. 明确接受变重的地方

这些地方只要手感需要，就允许加重。

### 4.1 Trigger 层

当前方案：

- `CGEventTap`

升级条件：

- `Right Option` 的 down/up 不稳定
- 输入法或系统设置导致冲突

升级动作：

- 增加 `IOHIDManager` 路径

决策：

`为了更稳，允许 trigger 层变重。`

### 4.2 ASR 层

当前方案：

- `qwen3-asr-flash`

升级条件：

- `trigger up -> inserted` 经常超过 `1.2s`
- 用户主观感觉明显慢于 竞品

升级动作：

- 替换为 `qwen3-asr-flash-realtime`

重要说明：

即使升级到 realtime，也不意味着 UI 必须显示 partial transcript。

可以继续保持 竞品 风格的极简交互，只让后台链路更快。

决策：

`为了更顺，允许 ASR 方案变重。`

### 4.3 插入层

当前方案：

- 通用 `AXUIElement`
- 剪贴板回退

升级条件：

- 某个高频 app 主路径反复失败

升级动作：

- 增加 app-specific insertion strategy

决策：

`为了更稳，允许 insertion 层出现 app 特化。`

### 4.4 HUD 层

当前方案：

- 轻量 AppKit HUD

升级条件：

- 当前 HUD 显示慢
- 抢焦点
- 动画或布局影响输入流

升级动作：

- 改成更原生、更薄的 AppKit 实现

决策：

`HUD 的任务是消除不确定感，不是展示设计。`

## 5. 各技术的 Team Leader 判断

### 5.1 `Swift + AppKit`

判断：

- 保留

理由：

- 更接近系统层
- 更适合输入工具

### 5.2 `Right Option` 作为开发键

判断：

- 保留

理由：

- 不和当前 竞品 的 `Fn` 冲突
- 能先验证整条输入链

说明：

- 只是开发键，不是架构核心假设

### 5.3 `AVAudioEngine`

判断：

- 保留

理由：

- 足够原生
- 适合做低摩擦输入工具

要求：

- 尽量预热
- 不要每次按下都从零初始化整套音频链

### 5.4 `qwen3-asr-flash`

判断：

- 先用

理由：

- 接入简单
- 当前范围足够

说明：

- 它不是最终不可变方案
- 一旦不够顺，直接升 realtime

### 5.5 `qwen3.5-flash`

判断：

- 保留

理由：

- clean-up 足够强
- 成本也低

要求：

- 强控 prompt
- 只读取最终文本
- 忽略 reasoning 输出

### 5.6 `AXUIElement`

判断：

- 保留且重点攻坚

理由：

- 这是最接近真实产品质量的层

### 5.7 `clipboard fallback`

判断：

- 保留

说明：

- 它是兜底，不是主体验

## 6. 升级触发条件

下面这些条件一旦出现，不再继续追求轻量实现。

### 6.1 触发不稳

条件：

- `Right Option` down/up 事件漏触发
- 实机误触发率高

动作：

- 补 `IOHIDManager`

### 6.2 总延迟偏高

条件：

- `trigger up -> inserted` 经常大于 `1200ms`
- 或明显慢于 竞品

动作：

- 升级 ASR 到 realtime

### 6.3 插入不稳

条件：

- 某个高频 app 主路径成功率明显不足

动作：

- 做 app-specific insertion strategy

### 6.4 clean-up 过度发挥

条件：

- 输出频繁改坏原意

动作：

- 进一步收紧 prompt
- 必要时降低 clean-up 复杂度

## 7. 当前执行原则

从现在开始，团队按这 4 条执行：

1. `手感优先于优雅`
2. `延迟优先于架构洁癖`
3. `主路径优先于通用性`
4. `失败不丢文本`

## 8. 结论

当前技术路线不推翻，但优先级已经调整：

- 不是追求最轻
- 而是追求最像 竞品
- 轻量只在不伤体验时保留

一句话：

`允许实现更重，不允许输入体验更钝。`
