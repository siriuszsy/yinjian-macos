# 音键 Vibe Coding 开发说明

最后更新：2026-04-11

## 1. 目的

这份文档是给使用 `vibe coding` 方式接手项目的人看的。

重点不是“怎么天马行空地改”，而是：

- 怎么先把项目编起来
- 怎么稳定地处理签名和权限
- 哪些地方不要乱动

## 2. 先做什么

开始前先做这 3 件事：

1. 读 [README.md](../README.md)
2. 读 [技术架构](./technical-architecture.md)
3. 读 [V1 系统设计](./v1-system-design.md)

不要一上来就改代码。

## 3. 最小编译路径

```bash
xcodebuild -project tinyTypeless.xcodeproj \
  -scheme tinyTypeless \
  -configuration Debug \
  -derivedDataPath .derived \
  build
```

先确保项目能编过，再改逻辑。

## 4. 不要乱动的地方

如果你只是为了继续迭代功能，以下内容不要顺手改：

- `Bundle Identifier`
- App 安装路径
- 签名方式
- 权限链路
- Trigger 默认策略
- ASR / cleanup 主链路

原因很直接：

- `辅助功能` 权限对 app 身份非常敏感
- 一旦你一边改签名、一边改路径、一边改 Bundle ID，很容易把 TCC 权限状态搞漂
- 这样你会分不清问题到底在代码、权限还是系统缓存

## 5. 当前稳定策略

当前项目有几条已经踩出来的经验：

### 5.1 安装路径要稳定

调试时建议最终运行路径固定在：

```text
/Applications/tinyTypeless.app
```

如果你频繁换运行路径，`辅助功能` 授权容易失效。

### 5.2 不要盲改触发方案

当前开发版默认触发键是：

`⌘ + ;`

这是为了绕开全局单键监听的权限复杂度。  
如果你没把权限和触发层研究透，不要直接切回 `Fn` 或右侧 `⌥`。

## 6. 苹果编译与本地签名

仓库里会保留脚本，但不会保留：

- 证书
- 私钥
- 密码

这部分的定位要明确：

- 它是苹果本地编译和调试辅助
- 它是为了让本地 `.app` 更稳定地处理权限
- 它不是产品功能的一部分

如果你在使用 `vibe coding`，可以把它理解成：

`一层苹果开发环境适配`

如果你要在本地做自己的签名：

1. 自己生成或导入本地开发证书
2. 自己设置环境变量
3. 再运行签名脚本

当前脚本要求的环境变量：

- `TINYTYPELESS_CODESIGN_KEYCHAIN_PASSWORD`
- `TINYTYPELESS_CODESIGN_P12_PASSWORD`

注意：

不要把这些密码写死回仓库。

## 7. 权限调试顺序

如果你在调试“为什么文字写不进输入框”，顺序必须是：

1. 先确认只有一个 app 实例在跑
2. 再确认 app 身份没有漂
3. 再确认 `辅助功能` 已授权
4. 最后才查 AX / 粘贴回退逻辑

不要反过来。

## 8. 当前推荐工作流

### 8.1 功能改动

1. 先改 `Domain`
2. 再改 `Application`
3. 再落 `Infrastructure`
4. 最后补 `Presentation`

### 8.2 系统问题排查

1. 先看日志
2. 先定根因
3. 再改代码

不要围着症状连续乱修。

## 9. 公开发布相关

如果你是在做 GitHub 源码发布，请坚持：

- 只发源码
- 只发文档
- 不发二进制
- 不发本地签名材料
- 不发 API Key

## 10. 一句话规则

如果你在 `vibe coding`：

`先把现状编通、跑通、看清楚，再做单点修改。`

不要一边改权限、一边改签名、一边改安装路径、一边改主链。
